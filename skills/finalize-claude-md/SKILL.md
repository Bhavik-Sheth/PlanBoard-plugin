# Finalize — CLAUDE.md Compilation Skill

Reads all PLANNER/ files and compiles a single CLAUDE.md at the project root.
Runs in the main thread (no subagent needed — this is a read + summarize pass).

---

## What to produce

Write `CLAUDE.md` at the **project root** (not inside PLANNER/).

CLAUDE.md is an execution-context document for a coding AI agent. It must be:
- Dense and accurate — the coding agent reads this on every prompt
- Under 300 lines — context bloat makes AI responses worse
- A summary, not a copy-paste — synthesize; never verbatim-copy sections

**The core rule: only include what a coding agent absolutely needs to write correct code for this project right now.**

If a piece of information is not directly needed to make implementation decisions, leave it out. CLAUDE.md is not a project overview, not a sales pitch, not a full spec mirror. Every line must earn its place by preventing a wrong implementation choice. When in doubt, cut it — the full PLANNER/ documents are always available for deeper reference.

---

## Required sections (in this order)

```
# CLAUDE.md
> Auto-generated by PlanBoard /finalize. Do not edit manually.
> Regenerate with /finalize after planning changes.

## Project Summary
[3–5 sentences from StructuredPlan.md: what it is, who uses it, core purpose]

## Tech Stack
[Table format: Layer | Technology | Version. Pull from TRD.md. Exact choices only — no alternatives.]

## Folder Structure
[Key directories and their purpose, one line each. Pull from ImplementationPlan.md slice "Touches" fields.]

## Coding Rules (Critical)
[Top 5–10 rules from Rules.md that affect day-to-day code. Bullets. Enforceable rules only — no platitudes.]

## Library-First Rule (always include — non-negotiable)
- Before writing any non-trivial logic, check whether a well-maintained library already solves it.
- If a library exists that handles the problem correctly, use it. Do not reimplement it.
- A solution that takes 5 lines with a library is always preferred over 200 lines written from scratch.
- Only write custom code when: no suitable library exists, the library's scope far exceeds what is needed, or a constraint explicitly forbids it.

## Hard Constraints
[Must-never-do list from PRD.md's Constraints section. Bullets. Absolute language.]

## Data Model (Summary)
[Key tables + primary columns only from Schema.md. Not full schema — orientation only.]

## Key API Endpoints
[Core endpoints from TRD.md. Format: METHOD /path — one-line purpose. Drop if file is absent.]

## Context Binding
Read the full planning documents in PLANNER/ for complete specs:
- PLANNER/PRD.md — product requirements, scope, user stories
- PLANNER/TRD.md — tech stack, architecture, API interfaces
- PLANNER/Schema.md — data model and relationships
- PLANNER/AppFlow.md — user flows and navigation (if present)
- PLANNER/DesignDecisions.md — architectural decision records (if present)
- PLANNER/Rules.md — coding conventions and AI behavior rules
- PLANNER/ImplementationPlan.md — implementation slices
- PLANNER/ARCHITECTURE_DIAGRAMS/ — ASCII diagrams (if generated)

The coding agent must bind all implementation choices to the documents above.
Do not deviate from the constraints, tech stack, schema, or rules defined therein.
```

## Rules

- **Only include what is absolutely required.** If a section contains information that doesn't directly affect how the coding agent writes code, cut it.
- **Never pad.** No introductions, no rationale, no historical context, no "this document explains..." — every line must be actionable.
- If a source file is absent or empty: write `[Not available — <filename> was not generated]` for that section
- Never exceed 300 lines total — if cutting is needed, drop "Key API Endpoints" first, then "Data Model (Summary)"
- Do not add sections beyond those listed above
- Do not include design rationale or decision history — that belongs in DesignDecisions.md
- Do not include user stories, success metrics, or acceptance criteria — that belongs in PRD.md
- Do not include full schema definitions — the summary table is enough for orientation; Schema.md has the rest
- Write clean Markdown — no surrounding code fences
